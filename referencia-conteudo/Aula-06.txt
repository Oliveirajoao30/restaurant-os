 •Projeto de Arquitetura

    q Arquitetura de Software refere-se à estrutura fundamental e
      organização de um sistema de software, incluindo componentes,
      interações e decisões de design de alto nível.
    q Ela define como os diferentes componentes do sistema se comunicam,
      interagem e cooperam para alcançar os objetivos do software.
    q Uma boa arquitetura pode melhorar a manutenibilidade, escalabilidade
      e extensibilidade do sistema.

 •Clean Code

    q Clean Code" (código limpo, em português) é um conceito fundamental
      na engenharia de software que se refere à prática de escrever código de
      programação de forma clara, organizada e compreensível. O objetivo do
      código limpo é tornar o software mais fácil de ler, entender, manter e
      estender. Escrever código limpo é essencial para o desenvolvimento de
      software de alta qualidade e eficaz.

 •Clean Code

    1.Legibilidade: O código deve ser fácil de ler e entender.
    Deve seguir convenções de nomenclatura consistentes e usar
    nomes de variáveis, funções e classes descritivos.
    2.Simplicidade: O código deve ser simples e direto. Evite
    complexidade desnecessária e mantenha funções e métodos
    pequenos e focados em uma única responsabilidade.
    3.Clareza: Evite ambiguidades e confusões. Comente o código
    quando necessário para explicar o raciocínio por trás das
    decisões de implementação.

    4. Consistência: Mantenha um estilo de codificação
    consistente em todo o projeto. Isso facilita a colaboração e a
    manutenção.
    5. Testabilidade: O código deve ser fácil de testar. Isso
    geralmente envolve a criação de testes automatizados para

    6. Reutilização: Promova a reutilização de código sempre

    7. Mantenha o código pequeno: Evite funções e classes

 •Clean Code
    q Nomes Significativos: Escolha nomes de variáveis, funções, classes e
      métodos que descrevam claramente sua finalidade e funcionalidade.
      Nomes significativos tornam o código mais legível.

    q Funções Pequenas e Concisas: O princípio da "função pequena" afirma
      que as funções e métodos devem ser curtos e focados em fazer uma
      única coisa. Funções pequenas são mais fáceis de entender e testar.

    q Comentários Significativos: Use comentários para explicar por que o
      código está fazendo algo, não o que está fazendo. Comentários
      excessivos podem ser indicativos de código ruim, mas comentários úteis
      podem esclarecer a lógica complexa.                              17

 •Clean Code
    q Formatação Consistente: Mantenha um estilo de formatação consistente
      em todo o código. Isso inclui a indentação, o uso de espaços em branco

      no código. Evite duplicar informações.

 •Clean Code
    q Baixo Acoplamento: Mantenha as partes do código independentes e

 •Princípio da Responsabilidade Única (Single Responsibility
 Principle - SRP)

    q Este princípio afirma que uma classe deve ter uma única razão para
      mudar. Em outras palavras, uma classe deve ter apenas uma
      responsabilidade ou função clara.
    q Isso promove a coesão, tornando as classes mais fáceis de entender,
      testar e manter.
    q Classes que aderem ao SRP são menos propensas a erros e mais
      reutilizáveis.

 •Princípio do Aberto/Fechado (Open/Closed Principle - OCP)

    q O OCP postula que as entidades de software (classes, módulos, funções,
      etc.) devem ser abertas para extensão, mas fechadas para modificação.
    q Isso significa que você pode estender o comportamento de uma
      entidade sem alterar seu código-fonte existente.
    q Isso é alcançado por meio de herança, interfaces e polimorfismo,
      permitindo que novas funcionalidades sejam adicionadas sem afetar o
      código existente.

 •Princípio da Substituição de Liskov (Liskov Substitution Principle -
 LSP)

    q O LSP afirma que objetos de uma classe derivada devem ser capazes de
      substituir objetos da classe base sem afetar a corretude do programa.
    q Em outras palavras, se uma classe herda de outra, ela deve ser capaz de
      substituir instâncias da classe pai sem causar erros ou comportamentos
      inesperados.
    q O LSP é essencial para o uso correto de herança e polimorfismo.

 •Princípio da Segregação de Interface (Interface Segregation
 Principle - ISP)

    q O ISP argumenta que as interfaces de uma classe não devem ser
      forçadas a implementar métodos que não são relevantes para ela.
    q Isso evita que classes tenham dependências desnecessárias e torna as
      interfaces mais específicas para as necessidades de suas
      implementações.
    q O ISP promove a coesão e a redução do acoplamento.

 •Princípio da Inversão de Dependência (Dependency Inversion
 Principle - DIP)

    q O DIP sugere que módulos de alto nível não devem depender de
      módulos de baixo nível, mas ambos devem depender de abstrações.
    q Além disso, abstrações não devem depender de detalhes. Isso incentiva
      o uso de interfaces ou classes abstratas como contratos entre
      componentes do sistema.
    q O DIP permite a criação de sistemas mais flexíveis e facilita a
      substituição de implementações concretas.
