O Problema que o Docker Resolve
“Mas na minha máquina funciona!”


 Um software depende de bibliotecas, versões e configurações do ambiente. Quando o ambiente muda,
 ele pode parar de funcionar.




        Máquina do dev                   Servidor de produção            Configuração manual
      Funciona perfeitamente no           Versões diferentes → erros,   Cada ambiente é montado “na
   computador de quem programou.        travamentos e horas perdidas.      mão”, difícil de repetir.




  A ideia central: empacotar a aplicação junto com tudo de que ela precisa, para rodar igual em
  qualquer lugar.
  É exatamente isso que o Docker faz.
O que é Docker?
Uma plataforma para criar e rodar aplicações em contêineres



   Definição simples
   Docker é uma ferramenta que coloca seu programa
   dentro de uma caixa padronizada — o contêiner —
   que roda da mesma forma em qualquer computador.




   A analogia do contêiner                                                         Consistente

   qualquer carga sem que o porto precise saber o que há                           lugar

   dentro, o contêiner Docker carrega sua aplicação
   pronta para rodar.

Por que o Docker surgiu?
Um pouco de contexto


 Antes dos contêineres, padronizar ambientes era caro e lento. O Docker (lançado em 2013) popularizou
 uma tecnologia que tornou isso simples e rápido.




               Antes                       Máquinas virtuais                        Contêineres
    Servidores físicos: 1 aplicação     Melhor uso do hardware, mas            Leves, rápidos e portáteis.
    por máquina, desperdício de          ainda pesadas e lentas para            Compartilham o sistema
              recursos.                            iniciar.                           operacional.




  Evolução: físico → virtual → contêiner. Cada passo trouxe mais eficiência e portabilidade.

Máquina Virtual vs. Contêiner
Por que o contêiner é mais leve e mais rápido



              Máquina Virtual                                             Contêiner (Docker)

                 App A | App B                                                 App A | App B


                  Bibliotecas (x2)                                        Bibliotecas (compartilhadas)


        Sistema Operacional convidado (x2)                                       Docker Engine


                    Hypervisor                                           Sistema Operacional anfitrião


           Sistema Operacional anfitrião




  Resultado: o contêiner não carrega um sistema operacional inteiro — por isso inicia em segundos e ocupa

Imagens Docker
O molde imutável da sua aplicação




   O que é uma imagem?
    • Pacote somente-leitura com o código, as
      bibliotecas e o sistema base.
    • É imutável: nunca muda depois de criada.                                imagem: minha-app
    • Identificada por nome e tag (ex.: nginx:1.25).
    • De uma imagem nascem vários contêineres iguais.


                                                                    cont. 1        cont. 2        cont. 3


                                                                  Mesma imagem, vários contêineres idênticos
                                                                         rodando ao mesmo tempo.




  Analogia: a imagem é a fôrma; os contêineres são os bolos feitos com ela.

 Cada instrução do Dockerfile cria uma camada. As camadas são empilhadas e reaproveitadas em cache.

   Sistema base                                               Por que camadas?
                                                                • Cache: o que não mudou é
                                                                  reaproveitado → builds mais rápidos.
   RUN apt-get install ...
   Dependências                                                 • Reuso: camadas comuns são
                                                                  compartilhadas entre imagens.
                                                                • Economia: menos espaço em disco e
   COPY . /app
   Código da aplicação                                            downloads menores.


   CMD ["python","app.py"]
   Comando de início




  Dica: coloque o que muda menos no topo do Dockerfile para aproveitar melhor o cache.
Registries e Docker Hub
Onde as imagens são guardadas e compartilhadas




   O que é um registry?
    • É um repositório de imagens, como um “GitHub
      de imagens”.
    • Docker Hub é o registry público mais usado.
                                                                         Registry (nuvem)
                                                                  docker pull   /   docker push
    • Empresas usam registries privados (AWS ECR,
      GitLab...).
    • Imagens prontas economizam muito tempo.
                                                              docker pull nginx
                                                              # baixa imagem
                                                              docker push user/app
                                                              # envia imagem




  pull baixa uma imagem do registry; push envia a sua imagem para lá.

Ciclo de Vida do Contêiner
Os estados de um contêiner




         Criado         →        Em execução       →           Parado         →         Removido
     docker create            docker start / run             docker stop                 docker rm




  Importante: parar um contêiner não o apaga. Os dados continuam até você removê-lo com docker rm.
Anatomia do Dockerfile
As instruções mais usadas




         FROM                                                   WORKDIR
    F    Imagem base de partida (ex.: python:3.12).
                                                         W      Define a pasta de trabalho dentro do contêiner.




         COPY                                                   RUN
    C    Copia arquivos do projeto para a imagem.
                                                          R     Executa comandos durante a construção (ex.:
                                                                instalar libs).



         EXPOSE                                                 CMD
    E    Documenta a porta que a aplicação usa.
                                                          C     Comando que roda quando o contêiner inicia.




  Ordem importa: instruções viram camadas — organize-as para aproveitar o cache.
Exemplo de Dockerfile
Uma aplicação Python simples




    FROM python:3.12-slim                 FROM Parte de uma imagem Python enxuta.
    WORKDIR /app
    COPY requirements.txt .
    RUN pip install -r requirements.txt   WORKDIR Trabalha dentro de /app.
    COPY . .
    EXPOSE 5000                           COPY + RUN Instala só as dependências
    CMD ["python", "app.py"]              primeiro (cache!).


                                          COPY . . Copia o restante do código.



                                          EXPOSE Informa a porta 5000.



                                          CMD Inicia a aplicação.

Volumes: Dados que Persistem
Como não perder dados ao remover um contêiner


 Por padrão, ao remover um contêiner os dados somem. Volumes guardam dados fora do contêiner, de
 forma permanente.


  Sem volume                                          Com volume
   • Dados ficam dentro do contêiner.                  • Dados ficam guardados na máquina anfitriã.
   • Removeu o contêiner → perdeu tudo.                • Contêiner vai e volta, dados permanecem.
   • Ruim para bancos de dados e uploads.              • Ideal para bancos, logs e arquivos.




   docker run -v meus-dados:/var/lib/dados minha-app
   # monta um volume chamado meus-dados

Docker Compose
Vários contêineres com um único comando


 Aplicações reais têm várias partes (site + banco + cache). O Compose descreve tudo em um arquivo e sobe
 com um comando só.




          Um arquivo                           Vários serviços                   Um comando
    docker-compose.yml descreve           Web, banco e cache definidos    docker compose up sobe tudo de
         todos os serviços.                         juntos.                         uma vez.




   docker compose up
   # sobe todos os serviços       |   docker compose down   # encerra
Exemplo de docker-compose.yml
Site + banco de dados juntos



                                        web
    services:                           O serviço do site, construído pelo Dockerfile
      web:                              local.
        build: .
        ports:
                                        ports
          - "8080:5000"

        depends_on:
          - db
      db:                               depends_on
        image: postgres:16              Garante que o banco suba antes do site.
        volumes:
          - dados:/var/lib/postgresql
                                        db
                                        Usa a imagem pronta do PostgreSQL.



                                        volumes

Boas Práticas
Imagens enxutas, builds rápidos e seguros




          Imagens leves                                           Cache esperto
          Use imagens base “slim” ou “alpine”.                    Copie dependências antes do código.




          .dockerignore                                           Uma tarefa
          Evite copiar lixo (node_modules, .git).                 Um processo principal por contêiner.




          Tags claras                                             Sem segredos
          Versione: app:1.0, não só latest.                       Nunca coloque senhas no Dockerfile.




  Regra de ouro: quanto menor e mais simples a imagem, mais rápida, segura e barata ela é.

            necessárias.                                             no código.




  Lembre-se: contêiner isola, mas não substitui boas práticas de segurança.

Erros Comuns de Iniciante
E como evitá-los




          Esquecer o .dockerignore                                Guardar dados sem volume
          Imagem fica enorme copiando arquivos inúteis.           Tudo é perdido ao recriar o contêiner.




          Usar sempre :latest                                     Senhas no Dockerfile
          Build de hoje pode diferir do de amanhã. Versione!      Qualquer um que ler a imagem vê o segredo.




          Imagem base gigante                                     Tudo em um contêiner
          Builds e downloads ficam lentos sem necessidade.        Misturar site + banco + cache dificulta manutenção.




  A boa notícia: todos esses erros somem com prática e as boas práticas que vimos.
