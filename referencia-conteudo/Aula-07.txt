 •Clean Code

    ❑ Clean Code" (código limpo, em português) é um conceito fundamental
      na engenharia de software que se refere à prática de escrever código de
      programação de forma clara, organizada e compreensível. O objetivo do
      código limpo é tornar o software mais fácil de ler, entender, manter e
      estender. Escrever código limpo é essencial para o desenvolvimento de
      software de alta qualidade e eficaz.

 •Clean Code

    1.Legibilidade: O código deve ser fácil de ler e entender. Deve seguir
    convenções de nomenclatura consistentes e usar nomes de variáveis,
    funções e classes descritivos.
    2.Simplicidade: O código deve ser simples e direto. Evite complexidade
    desnecessária e mantenha funções e métodos pequenos e focados em uma
    única responsabilidade.
    3.Clareza: Evite ambiguidades e confusões. Comente o código quando
    necessário para explicar o raciocínio por trás das decisões de
    implementação.

 •Clean Code

    4. Consistência: Mantenha um estilo de codificação consistente em todo o
    projeto. Isso facilita a colaboração e a manutenção.
    5. Testabilidade: O código deve ser fácil de testar. Isso geralmente envolve
    a criação de testes automatizados para garantir que o código funcione

    6. Reutilização: Promova a reutilização de código sempre que possível. Isso

    7. Mantenha o código pequeno: Evite funções e classes longas. O princípio

 •Clean Code
    ❑ Nomes Significativos: Escolha nomes de variáveis, funções, classes e
      métodos que descrevam claramente sua finalidade e funcionalidade.
      Nomes significativos tornam o código mais legível.

    ❑ Funções Pequenas e Concisas: O princípio da "função pequena" afirma
      que as funções e métodos devem ser curtos e focados em fazer uma
      única coisa. Funções pequenas são mais fáceis de entender e testar.

    ❑ Comentários Significativos: Use comentários para explicar por que o
      código está fazendo algo, não o que está fazendo. Comentários
      excessivos podem ser indicativos de código ruim, mas comentários úteis
      podem esclarecer a lógica complexa.                              17

 •Clean Code
    ❑ Formatação Consistente: Mantenha um estilo de formatação
      consistente em todo o código. Isso inclui a indentação, o uso de espaços
      em branco e a formatação de código.

    ❑ Evite Código Repetitivo: Evite duplicação de código, pois ela torna a
      manutenção mais difícil e aumenta as chances de erros. Extraia código
      repetitivo para funções ou classes reutilizáveis.

    ❑ Princípio DRY (Don't Repeat Yourself): Este princípio diz que cada peça
      de conhecimento ou lógica deve ser representada em apenas um lugar
      no código. Evite duplicar informações.

    ❑ Isso é alcançado por meio de herança, interfaces e polimorfismo,

 •Princípio da Substituição de Liskov (Liskov Substitution Principle -
 LSP)

    ❑ O LSP afirma que objetos de uma classe derivada devem ser capazes de
      substituir objetos da classe base sem afetar a corretude do programa.
    ❑ Em outras palavras, se uma classe herda de outra, ela deve ser capaz de
      substituir instâncias da classe pai sem causar erros ou comportamentos
      inesperados.
    ❑ O LSP é essencial para o uso correto de herança e polimorfismo.

 •Princípio da Inversão de Dependência (Dependency Inversion
 Principle - DIP)

    ❑ O DIP sugere que módulos de alto nível não devem depender de
      módulos de baixo nível, mas ambos devem depender de abstrações.
    ❑ Além disso, abstrações não devem depender de detalhes. Isso incentiva
      o uso de interfaces ou classes abstratas como contratos entre
      componentes do sistema.
    ❑ O DIP permite a criação de sistemas mais flexíveis e facilita a
      substituição de implementações concretas.

O que é TDD?
   Metodologia de desenvolvimento ágil criada por Kent Beck (eXtreme Programming, 2003).



   O ciclo básico é: escrever o teste → implementar → refatorar (Red → Green → Refactor).



   Nenhuma linha de código de produção é escrita sem um teste que a justifique.



   Os testes guiam o design do sistema, não apenas validam o código.



   Reduz bugs, melhora a cobertura de testes e aumenta a confiança nas mudanças.

class TestCalculoMedia(unittest.TestCase):

    def test_media_lista_normal(self):
self.assertEqual(calcular_media([1, 2, 3, 4, 5]), 3.0)

    def test_media_lista_vazia(self):             # edge case
        self.assertEqual(calcular_media([]), 0)

    def test_media_elemento_unico(self):         # edge case
        self.assertEqual(calcular_media([7]), 7.0)

    def test_media_valores_negativos(self):      # edge case
        self.assertEqual(calcular_media([-1, -2, -3]), -2.0)

    def test_media_valores_float(self):
        self.assertAlmostEqual(calcular_media([1.5, 2.5]), 2.0, places=5)
TDD vs Desenvolvimento Tradicional


Ordem          Código → Testes (talvez)       Testes → Código → Refactor


Design         Emergente e reativo            Guiado pelos testes


Bugs           Encontrados tarde (produção)   Capturados cedo (ciclo)


Refactor       Arriscado — sem rede           Seguro — testes protegem


Documentação   Separada e desatualizada       Os testes são a documentação


Cobertura      Geralmente baixa               Alta por design
                                           Resumo



        Escreva o teste —                     faça passar —              refatore.


•   TDD é uma disciplina — exige prática, mas os benefícios compensam.
•   O ciclo Red→Green→Refactor deve ser curto: minutos, não horas.
•   Testes são documentação viva do comportamento esperado do sistema.
•   Refatoração segura é só possível com boa cobertura de testes.
•   Comece pequeno: uma função, um teste, um ciclo por vez.

    ❑ BDD, ou Behavior-Driven Development (Desenvolvimento Orientado ao
      Comportamento), é uma abordagem de desenvolvimento de software
      que se concentra na compreensão e na descrição do comportamento
      do software a partir da perspectiva do usuário ou dos stakeholders. O
      BDD é uma extensão do TDD (Test-Driven Development) que coloca
      mais ênfase na linguagem natural e na colaboração entre diferentes
      partes interessadas, como desenvolvedores, testadores e especialistas
      em domínio.

Given um contexto · When uma ação · Then o resultado esperado.
O que é BDD?
   Criado por Dan North em 2003 como uma evolução natural do TDD, focando na linguagem de
   negócio.

   Testes são escritos em linguagem natural (Gherkin), compreensível por desenvolvedores, QA e
   stakeholders.


   Promove colaboração entre times técnicos e de negócio antes de qualquer linha de código.



   Cenários descrevem comportamentos do sistema do ponto de vista do usuário final.



   BDD é uma extensão do TDD: a diferença está no nível de abstração e no foco em comportamento.

  •   BDD é sobre comunicação — o código é consequência, não o objetivo.
  •   Gherkin serve de ponte entre linguagem técnica e de negócio.
  •   Cenários BDD são a especificação e a documentação ao mesmo tempo.
  •   Three Amigos (Dev + QA + PO) garantem que todos concordam antes de codificar.
  •   BDD e TDD são complementares: BDD no nível de aceitação, TDD no unitário.
